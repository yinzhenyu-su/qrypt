把它当成**“文件系统产品”而不是“网盘工具”**来迭代。核心原则：先跑通最小闭环，再逐层加复杂能力。

推荐迭代路线

1. 先做 MVP：只读挂载

目标不是加密，也不是同步，而是：

> 能把远端网盘目录挂载成本地目录，并能浏览、打开文件。

第一阶段只支持：

登录 / 配置网盘 token

列目录

读取文件

本地挂载

基础缓存

错误日志

不要一开始做上传、冲突解决、加密、断点续传。

---

2. 第二阶段：本地缓存层

网盘挂载最难的不是 UI，而是：

> 网络慢、不稳定，但本地文件系统要求“像本地文件一样快”。

所以第二阶段做缓存：

文件块缓存

元数据缓存

LRU 清理

缓存失效策略

离线只读模式

这层要独立出来，后面上传、加密都会依赖它。

---

3. 第三阶段：写入能力

再支持：

新建文件

修改文件

删除文件

重命名

移动目录

大文件分片上传

上传失败重试

这里要特别注意：不要直接边写边传网盘。

更稳的方案是：

本地写入 → 写入本地缓存/临时区 → 后台同步到网盘

---

4. 第四阶段：加密层

加密不要和网盘 API、挂载逻辑混在一起。

建议设计成独立层：

文件系统层
↓
缓存层
↓
加密/解密层
↓
网盘适配层

加密层负责：

文件内容加密

文件名加密

目录名加密

key 管理

salt / nonce / metadata

版本兼容

一开始可以只做文件内容加密，文件名加密放后面。

---

5. 第五阶段：可靠性与恢复

网盘挂载驱动最怕数据损坏，所以后期重点是：

操作日志 journal

同步状态表

失败任务队列

可恢复上传

冲突检测

文件校验 hash

本地缓存损坏修复

断电 / 断网恢复

---

架构建议

可以拆成 5 个核心模块：

Mount Driver 本地挂载层
VFS Core 虚拟文件系统抽象
Cache Manager 缓存管理
Crypto Layer 加密解密
Cloud Adapter 网盘 API 适配

每个模块只关心自己的事。

例如：

interface CloudAdapter {
list(path: string): Promise<FileEntry[]>
read(fileId: string, range?: Range): Promise<ArrayBuffer>
upload(path: string, data: Blob): Promise<void>
delete(path: string): Promise<void>
}

以后换百度网盘、阿里云盘、WebDAV、S3，只换 Adapter。

---

正确的开发节奏

每一轮迭代都问这 4 个问题：

1. 这次是否让挂载更稳定？

2. 是否降低数据丢失风险？

3. 是否有可复现测试？

4. 是否保持模块边界清晰？

不满足这几个，就先别加新功能。

---

最小可行路线

我建议顺序是：

只读挂载
→ 元数据缓存
→ 文件内容缓存
→ 分块读取
→ 本地写入缓存
→ 后台上传队列
→ 内容加密
→ 文件名加密
→ 冲突处理
→ 多网盘适配

一句话：先做“能稳定读”，再做“能安全写”，最后做“能可靠加密同步”。
